Chapters 6 and 7: Synchronization Tools and Examples

These chapters explain why shared data becomes dangerous, what a critical section is, and how mutexes and semaphores are used to protect correctness.

1. Race Condition

  • A race condition happens when concurrent processes or threads access shared data without proper synchronization.
  • The final result depends on the exact execution order.
  • The classic producer-consumer counter example shows how counter++ and counter-- can produce the wrong value if they overlap.
  • The core problem is not just sharing data, but sharing it without controlling who may update it and when.

2. Critical Section Problem

  • A critical section is the part of a program that accesses shared resources such as variables, files, tables, or buffers.
  • If one process is inside its critical section, another process must not enter a conflicting critical section at the same time.
  • A correct solution must satisfy mutual exclusion, progress, and bounded waiting.
  • These three conditions appear often in exam definitions, so memorize what each one really means.

Peterson's Solution

  • Classic software solution for two processes only.
  • Uses a shared turn variable and a flag[] array.
  • It is useful conceptually because it shows how mutual exclusion can be achieved in software.
  • Modern hardware may reorder instructions, so Peterson's solution is not fully reliable there.

Hardware Support

  • Modern systems provide atomic hardware instructions.
  • Important examples are test-and-set and compare-and-swap.
  • These instructions are uninterruptible, so they are used as building blocks for synchronization tools.

3. Mutex Locks

  • A mutex lock is a software tool that provides mutual exclusion.
  • A thread or process must acquire() the lock before entering the critical section and release() it afterward.
  • If a process keeps checking the lock in a loop, that is busy waiting.
  • A busy-wait mutex is often called a spinlock.
  • Spinlocks waste CPU time, but they can still be useful when the waiting period is expected to be very short.

4. Semaphores

  • A semaphore is an integer synchronization variable accessed only through atomic wait() and signal() operations.
  • Binary semaphore: value is only 0 or 1, so it behaves similarly to a mutex.
  • Counting semaphore: value can be greater than 1, so it can represent multiple resource instances.
  • Semaphores can be implemented with busy waiting, or with blocking queues to avoid wasting CPU cycles.

5. Semaphore Examples

  • Bounded buffer usually uses three semaphores: mutex, full, and empty.
  • Readers-writers separates readers, who may often share access, from writers, who need exclusive access.
  • In the first readers-writers problem, readers have priority. In the second, writers have priority.
  • Both priority choices can create starvation if the design is not careful.

6. Common Mistakes and Deadlock Risk

  • Calling signal before the matching wait can break synchronization logic.
  • Calling wait twice on the same mutex can block the process itself.
  • Forgetting wait causes race conditions; forgetting signal can make others wait forever.
  • If two processes acquire resources in opposite order, deadlock can happen.
  • This is why synchronization is not only about speed or ordering, but also about correctness and safety.
Clean final-exam story: race condition → critical section problem → mutex / semaphore → synchronization examples → deadlock risk when tools are misused.

Atomic Variables

  • Atomic variables are special data types that allow updates without interruption.
  • They are useful for simple shared values such as counters and flags.
  • They are often built using hardware primitives like compare-and-swap.

Busy Waiting vs Blocking

  • Busy waiting wastes CPU time because the process keeps checking in a loop.
  • Blocking suspends the process and moves it to a wait queue until wakeup.
  • Blocking semaphores are better when the wait may be long.

7. Bounded Buffer Details

  • The bounded-buffer problem uses mutex for mutual exclusion, empty to count free slots, and full to count occupied slots.
  • The producer must usually do wait(empty) then wait(mutex) before inserting an item.
  • The consumer must usually do wait(full) then wait(mutex) before removing an item.
  • After the critical section, the producer signals mutex then full, while the consumer signals mutex then empty.
  • The order matters because it preserves both correctness and proper counting of available slots.

8. Readers-Writers Details

  • Readers may read together as long as no writer is writing.
  • Writers need exclusive access because writing while another thread reads or writes can corrupt the shared data.
  • In the readers-preference solution, the first reader locks writers out and the last reader lets writers in again.
  • The variable read_count tracks how many readers are currently inside.
  • Readers-preference can starve writers, while writers-preference can starve readers.

9. Semaphore Usage Patterns

  • A binary semaphore initialized to 1 can be used like a mutex for mutual exclusion.
  • A counting semaphore initialized to n can track n identical resources.
  • Semaphores can also enforce execution order, such as making sure statement 1 happens before statement 2.
  • This is why semaphores are not only for locking, but also for general synchronization.

10. Semaphore Implementation Without Busy Waiting

  • To avoid busy waiting, each semaphore can keep a waiting queue of blocked processes.
  • If wait() cannot continue, the process is blocked and moved to that queue.
  • When another process executes signal(), one blocked process can be woken up and returned to the ready queue.
  • This approach wastes less CPU time than spin-based waiting.
Slide-level difference to remember: busy wait = loop on CPU, blocking wait = sleep in queue.

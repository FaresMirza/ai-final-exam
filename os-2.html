Chapter 2: Operating-System Structures

This chapter is about how the OS is built internally, what services it exposes, and how programs safely ask the kernel to do privileged work.

OS Components

  • Process management
  • Memory management
  • File-system management
  • I/O and device management
  • Networking, security, and user interface

OS Services

  • User interface
  • Program execution
  • I/O operations and file access
  • Communication and error detection
  • Protection and security

1. Components vs Services

  • Components are the internal subsystems of the operating system.
  • Services are the functions that users and programs can actually use.
  • A good short distinction is: components are inside the kernel, while services are what the kernel provides.
  • This distinction matters because exam questions often ask whether something is an internal OS responsibility or an exposed service.

2. User Interfaces

  • CLI means the user types commands into a shell such as Bash or Zsh.
  • GUI uses windows, icons, buttons, and pointer interaction.
  • Touch interface is common in phones and tablets where gestures replace a mouse.
  • Batch systems run jobs without direct interaction while the job is executing.
  • CLI is often preferred for scripting, automation, and remote management.

3. System Calls

  • A system call is the mechanism a user program uses to request a privileged kernel service.
  • Typical flow: program calls an API, the API invokes a system call, the CPU switches to kernel mode, the OS performs the work, then returns.
  • Common examples are open, read, write, fork, exec, wait, and exit.
  • System calls are needed because user programs do not have direct permission to access hardware or protected kernel state.
Easy answer: a system call is the controlled door from user mode into kernel mode.

4. Main Categories of System Calls

Core control categories

  • Process control
  • File management
  • Device management

Support categories

  • Information maintenance
  • Communication
  • Protection

5. OS Structure Designs

StructureMain IdeaMain Tradeoff
SimpleVery little separationFast but weak protection and poor organization
MonolithicOne large kernel in one address spaceFast but hard to maintain and isolate faults
LayeredOS is divided into layersCleaner design but extra overhead
MicrokernelMinimal kernel, more services in user spaceSafer and modular but slower due to messaging
ModularCore kernel plus loadable modulesFlexible and efficient, but module bugs still matter
HybridMix of multiple design ideasPractical but more complex internally

6. Important Comparisons

  • Monolithic kernel: high performance because services run together in kernel space, but one serious bug can affect the whole system.
  • Microkernel: better isolation and extensibility because more services live in user space, but communication overhead is higher.
  • Modular systems: keep good performance while allowing features such as drivers to be loaded dynamically.
  • Hybrid systems exist because real operating systems usually want both speed and flexibility.

API vs System Call

  • An API is the programmer-friendly interface used in code.
  • A system call is the lower-level kernel request that actually crosses into kernel mode.
  • Programs often call a library function first, and that library function performs the system call.

Passing Parameters

  • Parameters can be passed in registers for speed.
  • They can also be placed in a memory block and its address is passed to the kernel.
  • Some systems use the stack to pass arguments too.

7. System Programs and User View

  • System programs are not the kernel itself, but they provide a convenient environment for users and developers.
  • Examples include file tools, status tools, compilers, shells, text editors, communication utilities, and background services.
  • The user usually experiences the OS through these tools more than through the kernel directly.
  • This is why books often separate OS services from system programs.

8. What to Remember About System Calls

  • Process control calls handle creating, executing, waiting for, and ending processes.
  • File-management calls handle create, delete, open, close, read, write, and reposition.
  • Device-management calls request devices, release them, read from them, and write to them.
  • Communication calls let processes exchange information either through shared memory or message passing.
  • Protection calls check permissions and help control who may access which resource.
Fast exam view: API is what the programmer writes, system call is what the kernel receives.

9. Communication Models in System Services

  • Message passing lets the OS transfer packets or messages between processes.
  • Shared memory lets processes communicate through a common memory region after setup.
  • Message passing is cleaner conceptually, while shared memory is usually faster after it is established.
  • These models appear here because communication is one of the major operating-system services.

10. Quick Structure Recall

  • Simple structure: very little separation, fast but unsafe.
  • Monolithic: one large kernel, high performance, weak fault isolation.
  • Microkernel: minimal kernel plus user-space services, safer but slower.
  • Modular: kernel can load features dynamically without rebuilding or rebooting.
  • Hybrid: practical mix of multiple structure ideas used by real systems.

Chapter 4: Threads and Concurrency

Threads are lighter than processes, but they introduce shared-state risks. This chapter is mostly about that tradeoff and about how multicore execution changes performance.

1. What a Thread Is

  • A thread is the basic unit of CPU execution inside a process.
  • Each thread has its own thread ID, program counter, registers, and stack.
  • Threads in the same process share code, data, heap, and open files.
  • A traditional process with one thread is called a single-threaded process, while a process with several threads is multithreaded.

Why Threads?

  • Responsiveness for interactive applications.
  • Resource sharing inside one process.
  • Lower creation and switching cost than processes.
  • Scalability on multicore systems.

Process vs Thread

  • Process: heavier and more isolated.
  • Thread: lighter and shares process resources.
  • A thread failure can damage the entire process more easily.

2. Concurrency vs Parallelism

  • Concurrency means multiple tasks make progress over time, even on one core.
  • Parallelism means tasks truly run at the same time on multiple cores.
  • Data parallelism means doing the same operation on different parts of the data.
  • Task parallelism means different tasks run in parallel, possibly on the same or different data.

3. User Threads vs Kernel Threads

TypeMain Point
User threadsManaged in user space, cheaper, but weaker direct multicore control
Kernel threadsManaged by the OS, support real multicore parallelism, but with more overhead
Key exam idea: kernel threads are what let the OS schedule work directly on multiple processors.

4. Thread Mapping Models

  • Many-to-one: many user threads map to one kernel thread, so there is no true parallelism.
  • One-to-one: each user thread maps to one kernel thread, so true parallelism is possible.
  • Many-to-many: many user threads map to a smaller or flexible set of kernel threads.
  • The main exam question is usually which model allows parallel execution and which one blocks the whole group when one thread blocks.

5. Thread Libraries and Pthreads

  • pthread_create() creates a new thread.
  • pthread_join() waits for a thread to finish and helps release its resources correctly.
  • pthread_exit() terminates the current thread cleanly.
  • If the main thread exits too early, other threads may stop before finishing their work.

6. Main Risk with Threads

  • Threads share memory, so unsynchronized updates can create race conditions.
  • If two threads update the same variable at nearly the same time, the result may differ from run to run.
  • This is why synchronization tools such as mutexes and semaphores become necessary.
  • The power of threads is speed and responsiveness, but the cost is harder correctness.
Big takeaway: threads are efficient because they share resources, but that same sharing is exactly what creates concurrency bugs.

Chapter 3: Processes

This chapter covers program vs process, memory layout, process states, PCB, creation and termination, and the first big IPC ideas.

Program vs Process

  • A program is passive and stored on disk.
  • A process is a program currently executing in memory.
  • A process has a PID, state, CPU context, and resources.

Process Memory Layout

  • Text: executable code.
  • Data: global and static variables.
  • Heap: dynamic memory, usually grows upward.
  • Stack: function calls and local variables, usually grows downward.

1. Process Concept

  • A process is the basic unit of execution in most operating systems.
  • It includes the program code plus the current activity such as program counter, registers, stack, and allocated resources.
  • Several processes can be different executions of the same program, each with its own state.
  • This is why one browser program can create many separate processes or tabs.

2. Process States

  • Main states are new, ready, running, waiting, and terminated.
  • Ready means the process is in memory and waiting for CPU time.
  • Waiting means the process is blocked for I/O or some event.
  • Interrupts, I/O completion, and scheduler decisions move the process from one state to another.

3. PCB and Scheduling Queues

  • The Process Control Block (PCB) stores process state, PID, program counter, CPU registers, scheduling information, memory information, accounting information, and I/O status.
  • The ready queue contains processes waiting for the CPU.
  • The wait queue contains processes waiting for I/O or another event.
  • Context switching works by saving the old process state in its PCB and restoring the new process state from its PCB.

Context Switch

  • The OS saves one process state and restores another.
  • This is pure overhead because useful user work is not being done during the switch.
  • More complex systems usually have higher context-switch cost.

I/O-Bound vs CPU-Bound

  • I/O-bound: spends more time waiting than computing.
  • CPU-bound: spends more time computing than waiting.
  • This matters later in CPU scheduling decisions.

4. Process Creation

  • A parent process can create child processes, forming a process tree.
  • In UNIX, fork() creates the child, exec() replaces its program image, and wait() allows the parent to wait.
  • Parent and child may execute concurrently, or the parent may wait for the child to finish first.
  • Copy-on-write means memory is not physically copied until one side actually writes to it.

5. Process Termination

  • exit() ends a process normally and releases its resources.
  • A child becomes a zombie when it has finished but its parent has not yet collected the exit status.
  • A child becomes an orphan when the parent terminates first, then another system process adopts it.
  • Too many zombies are bad because they occupy entries in the process table.
Memory trick: Zombie = dead child, alive parent. Orphan = alive child, dead parent.

6. IPC Basics

  • IPC exists because processes often need to cooperate and exchange data.
  • Shared memory is very fast but needs synchronization.
  • Message passing is conceptually cleaner because the OS manages the communication.
  • Common reasons for IPC include information sharing, speedup, modular design, and convenience.

Stack vs Heap

  • The stack stores function calls, parameters, local variables, and return addresses.
  • The heap stores dynamically allocated objects that may live longer than one function call.
  • Stack allocation is usually automatic and fast, while heap use is more flexible but needs explicit management or garbage collection.

Direct vs Indirect Communication

  • Direct communication names the sender or receiver explicitly.
  • Indirect communication sends messages through a mailbox or port.
  • Mailboxes make communication more flexible because processes do not have to know each other directly.

7. More About Process Creation

  • Parent and child may share all resources, share only some resources, or share none, depending on the OS design.
  • In UNIX, fork() returns twice: once in the parent with the child PID, and once in the child with return value 0.
  • After fork(), the child often runs exec() to replace its memory image with a different program.
  • If the parent uses wait(), it can pause until the child finishes and then collect the exit status.
  • Without exec(), parent and child start from the same code location but continue as separate processes.

8. More About Process Termination

  • A process may terminate normally, terminate because of an error, or be aborted by another process or the OS.
  • Cascading termination means some systems terminate child processes automatically when the parent terminates.
  • A zombie has finished execution but still keeps a process-table entry until the parent reads its status.
  • An orphan keeps running after its parent has ended, so another system process adopts it.
  • These terms are common exam vocabulary because they test whether you understand parent-child cleanup behavior.

9. IPC Design Details

  • Shared memory is usually faster because after setup, processes read and write the same region directly.
  • Message passing is often easier to reason about because the kernel controls send and receive operations.
  • Communication may be blocking or non-blocking, depending on whether send/receive wait.
  • Buffers may be zero-capacity, bounded-capacity, or unbounded-capacity, and this changes whether processes must wait.
  • The bounded-buffer problem is a classic example showing why IPC often needs synchronization too.

PCB Main Fields

  • Process state and process ID.
  • Program counter and CPU registers.
  • CPU-scheduling information and memory-management information.
  • Accounting information and I/O status information.
  • The PCB lives in kernel memory because the OS must protect and update it safely.

Threads Inside a Process

  • A process may contain one thread or multiple threads of execution.
  • Threads of the same process share code, data, and open resources.
  • Each thread still needs its own program counter, register set, and stack.

10. UNIX Process Flow

  • fork() creates a new child process by duplicating the parent process structure.
  • After fork(), the parent receives the child PID, while the child receives return value 0.
  • exec() then replaces the child memory image with a new program.
  • wait() lets the parent stop until the child finishes and returns its status.
  • This sequence is one of the most common process-operation questions in OS exams.

11. Shared Memory vs Message Passing

  • Shared memory is faster after setup because the OS is no longer in the middle of every data transfer.
  • Message passing is easier to manage conceptually because the OS controls the send and receive operations.
  • Shared memory usually needs explicit synchronization, while message passing already includes a communication structure.
  • Shared memory is often preferred for high-speed local exchange, while message passing is common in distributed systems.

12. Key Process System Calls

  • fork(): create child process.
  • exec(): replace current program image.
  • wait(): wait for child and collect termination status.
  • exit(): terminate process and release resources.
  • Together, these calls describe most of the process lifecycle in UNIX-style systems.
Fast flow to remember: create with fork, replace with exec, synchronize with wait, finish with exit.

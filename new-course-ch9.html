Chapter 9

Case Studies in Modern Software Engineering

Why These Case Studies Matter

Chapter 9 ties the whole course together by showing how large companies apply the ideas from earlier chapters in the real world.

  • Netflix highlights microservices, resilience, observability, and CI/CD.
  • Spotify shows DevOps culture, organizational design, DDD, and event streaming.
  • Google/TensorFlow demonstrates AI-driven software engineering, MLOps, ethics, and sustainability.
  • Amazon/AWS Lambda shows event-driven and serverless computing with cost and energy efficiency.

Netflix Background and the Monolith Problem

  • Before 2010, Netflix had a successful DVD-by-mail service plus a growing streaming business.
  • The system ran as a large proprietary monolith in Netflix's private data center.
  • A major database corruption event caused a multi-day outage, proving the monolith was too risky.
  • Netflix committed to moving fully to AWS + distributed microservices.

Netflix Architecture: High Cohesion, Low Coupling

  • Each service focuses on one business capability such as billing, search, or recommendations.
  • Services use polyglot persistence, choosing the database technology that fits their own needs.
  • Teams can develop, test, and deploy services independently.
Ingress point

Client traffic first hits the API gateway layer, which handles routing and access before forwarding requests to many smaller services.

Hystrix and Graceful Failure

Netflix popularized the circuit breaker pattern through Hystrix.

  • If a dependency starts failing too often, the circuit opens.
  • Requests are short-circuited to a fallback rather than waiting for repeated timeouts.
  • This prevents cascading failures across the system.
Example

If personalization fails, Netflix can still show a generic recommendations list so the app remains usable.

Chaos Engineering at Netflix

  • Chaos Engineering means injecting failure on purpose to test resilience.
  • Chaos Monkey randomly terminates production instances during working hours.
  • The idea is simple: if your system breaks under controlled failure, it was never truly resilient.
Lesson

The only way to trust resilience is to keep testing it under realistic failure conditions.

Netflix CI/CD and Observability

Delivery

  • Spinnaker manages continuous delivery
  • Pipeline stages include build, test, canary, rollout, rollback
  • Supports very frequent deployments

Observability

  • Metrics for health and latency
  • Centralized logging for debugging
  • Distributed tracing across many services

Spotify's Scaling Problem

  • Spotify had to serve huge catalogs and massive numbers of users with low-latency personalization.
  • Traditional functional departments created slow delivery and heavy dependencies.
  • The challenge was not only technical architecture, but also how to organize people for speed and autonomy.

The Squad Model

  • Squads are small cross-functional teams focused on one product area.
  • Tribes group related squads.
  • Chapters connect people with the same specialty.
  • Guilds are wider communities of practice across the company.
Core idea

Spotify scaled by aligning team structure with product ownership, not just with technical specialties.

DevOps Culture: You Build It, You Run It

  • Squads own the full lifecycle: development, deployment, operation, and maintenance.
  • Self-service infrastructure reduces waiting on external teams.
  • Spotify emphasizes blameless postmortems so incidents improve the system, not punish individuals.

DDD and Event Streaming at Spotify

  • Spotify aligns microservices with bounded contexts such as user accounts, playback, or recommendations.
  • Each context owns its own terms and data model, reducing confusion.
  • Services communicate heavily through asynchronous event streams such as Kafka.
Remember

Spotify is the clearest example of organization design reinforcing software architecture.

Spotify CI/CD and Feature Flags

  • Continuous integration runs automated testing on every commit.
  • Continuous delivery uses canary deployments to release to small user groups first.
  • Feature flags separate code deployment from feature release, enabling A/B testing and instant rollback.

Google and the ML Revolution

  • Google's products depend heavily on machine learning for ranking, targeting, and recommendations.
  • Thousands of models and many teams created complexity beyond normal DevOps tooling.
  • Google responded by adding stronger engineering rigor through MLOps and frameworks like TensorFlow Extended (TFX).

MLOps and Continuous Training

  • MLOps treats models, data, and code as first-class artifacts.
  • Its pipeline includes data validation, model training, model validation, deployment, and monitoring.
  • A key addition beyond DevOps is Continuous Training (CT) when data drift or model decay appears.

DevOps

  • Focus: application code and binaries
  • Pipeline: CI/CD
  • Goal: fast stable deployment

MLOps

  • Focus: code, training data, model weights
  • Pipeline: CI/CD + CT
  • Goal: stability plus model accuracy

Sustainability in ML Systems

  • Large ML models consume huge amounts of compute and energy during training.
  • Google improves efficiency through TPUs and highly optimized data-center design.
  • Engineers should prefer smaller or more efficient models when accuracy gains are marginal.
Course link

This case connects AI-driven engineering directly to the sustainability chapter.

Ethics, Bias, and Explainability in ML

  • ML models can reproduce or amplify harmful bias if trained on skewed data.
  • Google uses Fairness Indicators to measure disparities across groups.
  • Explainable AI tools such as LIME and SHAP help explain why a prediction happened.
  • Ethical testing is part of engineering responsibility, not an afterthought.

Amazon's Problem: Idle Compute and Waste

  • Even inside AWS, many workloads were over-provisioned on EC2 instances.
  • Intermittent or spiky workloads caused idle cost and wasted energy.
  • AWS needed a model that could run code for milliseconds without requiring full server ownership.

AWS Lambda and the Serverless Shift

  • AWS Lambda launched the mainstream Function-as-a-Service model.
  • Developers write functions while the provider handles servers, patching, and scaling.
  • Billing is based on actual execution time rather than provisioned capacity.
Why it mattered

Serverless changed the economics of cloud software by allowing scale up and scale down to near-zero overhead.

Event-Driven Serverless Flow

  • A trigger such as an S3 upload, DynamoDB change, or API Gateway request invokes a stateless function.
  • The function runs only for the required duration, then stops.
  • The output can trigger the next event, making serverless a strong match for EDA.

Serverless Sustainability and Efficiency

  • Serverless avoids always-on dedicated servers for low-utilization workloads.
  • Shared infrastructure improves resource utilization and reduces idle energy waste.
  • This benefits both the user through lower cost and the environment through higher efficiency.

Cross-Case Lessons

Architecture lessons

  • Netflix: microservices + resilience
  • Spotify: DDD + event streaming + team alignment
  • Google: MLOps for model-heavy systems
  • Amazon: serverless + event-driven execution

Broader lessons

  • DevOps is technical and organizational
  • Observability and testing build trust
  • Ethics and sustainability are operational concerns
  • Modern SE is about fit, not one universal pattern

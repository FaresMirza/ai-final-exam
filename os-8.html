Chapter 8: Deadlocks

Deadlock questions usually test definitions, the four conditions, graph reading, and the difference between prevention, avoidance, and recovery.

1. What Deadlock Means

  • Deadlock is a situation where every process in a set is waiting for an event that can only be caused by another process in the same set.
  • The usual picture is: each process holds one resource and waits for another.
  • Mutex locks and semaphores are very common practical sources of deadlock.
  • If the waiting chain never breaks, the processes remain blocked forever.

2. System Model

  • The system contains resource types such as CPU cycles, memory, I/O devices, files, and synchronization objects.
  • Each process uses resources in this order: request, then use, then release.
  • If a requested resource is unavailable, the process waits.
  • Deadlock becomes possible when waiting processes keep resources that others need.

3. Four Necessary Conditions

  1. Mutual exclusion: only one process can use the resource at a time.
  2. Hold and wait: a process holds at least one resource while requesting more.
  3. No preemption: resources cannot simply be taken away.
  4. Circular wait: there is a cycle of processes waiting for each other.
All four conditions must hold together for deadlock to occur.

4. Resource-Allocation Graph

  • Processes/threads and resource types are shown as graph nodes.
  • Request edge: process → resource.
  • Assignment edge: resource → process.
  • If there is no cycle, there is no deadlock.
  • If each resource type has only one instance, a cycle means deadlock.
  • If resource types have several instances, a cycle means deadlock is possible, but not guaranteed in every case.

5. Deadlock Prevention and Avoidance

  • Prevention means breaking at least one of the four necessary conditions.
  • Examples: force all resources to be requested in advance, allow preemption in some cases, or impose a fixed ordering on resource requests.
  • Avoidance means checking every request and allowing it only if the system stays in a safe state.
  • Safe state means there is some order in which all processes can still finish.
  • Unsafe state does not mean deadlock already happened, but it means deadlock can happen.

6. Recovery from Deadlock

  • One recovery option is to abort all deadlocked processes.
  • Another is to abort them one by one until the cycle breaks.
  • A third option is to preempt resources from some process and give them elsewhere.
  • Important recovery concerns are victim selection, rollback cost, and starvation.
Quick distinction: prevention stops the conditions, avoidance checks safe state, and recovery fixes the problem after deadlock happens.

Semaphore Deadlock Example

  • If process P0 does wait(S) then wait(Q), while P1 does wait(Q) then wait(S), both can block forever.
  • Each process holds one semaphore and waits for the other one.
  • This is a direct practical example of circular wait.

Cycle Facts

  • No cycle in the resource-allocation graph means no deadlock.
  • With one instance per resource type, a cycle means deadlock.
  • With multiple instances, a cycle means deadlock is possible, not guaranteed.

7. Prevention by Breaking Conditions

  • Breaking mutual exclusion is possible only for sharable resources, so it is not a general solution.
  • Breaking hold and wait can be done by forcing a process to request all resources at once or only request when holding none.
  • Breaking no preemption means resources may be taken back if the next needed resource cannot be granted.
  • Breaking circular wait is often the most practical method: impose a fixed order on all resource types and force requests to follow that order.

8. Safe State and Avoidance Notes

  • A safe state means there exists some safe sequence in which all processes can still finish.
  • An unsafe state does not mean the system is already deadlocked, but it means deadlock may occur later.
  • Avoidance needs more prior knowledge than prevention, such as the maximum resources each process may request.
  • For one instance per resource type, a modified resource-allocation graph can be used.
  • For several instances, the classic reading topic is the Banker's algorithm.

9. Recovery Decision Factors

  • If the system aborts processes one by one, it must decide the victim carefully.
  • Important factors include process priority, how long the process has run, how many resources it holds, how many more it still needs, and whether it is interactive or batch.
  • If the system uses resource preemption, it must also consider rollback cost and repeated victim selection.
  • This is where starvation can appear again, because the same process might be chosen as victim too often.

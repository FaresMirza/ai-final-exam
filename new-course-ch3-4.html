Chapter 3

DevOps and Code Management

The Traditional Support Model

Traditionally, three separate teams handled software lifecycle stages:

  • Development team — wrote a "final" version and handed it over.
  • Release team — built the release version, tested, and prepared documentation before shipping.
  • Support team — provided customer support after release.
Remember — the problem

Separate teams = inevitable delays and overheads. Hand-offs slow everything down.

DevOps (Development + Operations)

An alternative approach that merges development, deployment, and support into one multi-skilled team. Three factors drove its rise:

  • Agile reduced dev time, exposing the release process as the new bottleneck.
  • Amazon re-engineered around services where one team owned a service end-to-end — and publicised the reliability gains.
  • Cloud delivery made it possible to release software as a service — no more physical media or downloads.

The Three DevOps Principles

  • Everyone is responsible for everything — joint responsibility for dev, delivery, and support.
  • Everything that can be automated should be automated — minimal manual involvement in deployment.
  • Measure first, change later — collect data, then use it to inform decisions about process and tool changes.

Benefits of DevOps

Speed & Risk

  • Faster deployment — fewer comm delays
  • Reduced risk — small increments, fewer feature collisions
  • Faster repair — no waiting for "the right team"

People

  • Happier, more productive teams
  • Lower turnover
  • Shared ownership of outcomes

Code Management

A set of software-supported practices for managing an evolving codebase — tens of thousands of lines, hundreds of files, dozens of libraries.

  • Prevents developers from overwriting each other's changes.
  • Lets you create and track different product versions.
  • Makes it easy to build the executable and run automated tests.

The Alice & Bob Story

Cautionary tale

Alice fixes a tax-return bug in TaxReturnPreparation. Bob — unaware — had taken a copy before her fix and saves his own changes, overwriting Alice's fix. Tests miss the intermittent bug. Customers' tax returns aren't filed. Fines hit. FinanceMadeSimple fails. Both lose their jobs.

Remember

Without proper code management, parallel work silently destroys itself. This is the problem source control exists to solve.

Code Repository & Features

All code management systems centre on a shared repository that stores source files, configs, build scripts, libraries, and history.

  • Code transfer — pull to local workstation, push back.
  • Version storage & retrieval — many versions; any can be fetched.
  • Branching & merging — parallel work, then reconcile.
  • Version info — who changed what, when, and why.
  • Storage management — efficient diffs, not full copies.

Git — Distributed Version Control

Created by Linus Torvalds in 2005 to manage the Linux kernel. The breakthrough: instead of keeping only the files you're editing, Git clones the entire repository onto every user's machine.

Resilience

  • Every clone is a backup
  • Work offline
  • Restore the shared repo from any clone

Speed & Flexibility

  • Commits are local — no network
  • Experiment privately without exposing work

Branching and Merging

A branch is an independent stand-alone version created when someone wants to change a file. Changes from different branches are then merged into a shared branch.

  • Made a mistake? Revert to the master file.
  • Committed mid-work? Roll back to any earlier commit.
  • Done and tested? Merge into the master branch.
Remember

The repo guarantees branch changes cannot overwrite master files without an explicit merge — exactly the protection Alice and Bob lacked.

DevOps Automation — Four Aspects

  • Continuous integration (CI) — every commit to master triggers a build and test.
  • Continuous delivery — the new version is tested in a simulated production environment.
  • Continuous deployment — every master change is auto-released to users.
  • Infrastructure as code (IaC) — machine-readable models build the execution platform automatically.

Continuous Integration

Every push to the shared repo triggers an integration server to build and test a new version.

  • GET source from code management
  • COMPILE & BUILD using libraries, config, and DB files
  • TEST against executable tests → deployable system
Don't break the build

"Breaking the build" = pushing code that fails system tests. Use the integrate-twice rule: build & test locally before pushing to the project repo.

System Building & Dependencies

For CI to feel fast, builds must be incremental — only recompile parts that changed. Automated build tools follow a dependency model using file timestamps:

  • Compiled file newer than source → do nothing.
  • Source newer than compiled → recompile.
  • Any dependency (e.g. Classdef) changed → recompile the dependent file.

Continuous Delivery vs Deployment

Continuous Delivery

  • Configure a staged test environment mirroring production
  • Install & run acceptance tests (functionality, load, performance)
  • Software is ready to ship

Continuous Deployment

  • Install on production servers
  • Switch traffic to the new version
  • Software is shipped — automatically, on every change
Remember

Continuous deployment is only practical when the product is delivered as a cloud service.

Benefits of Continuous Deployment

  • Reduced costs — automated pipelines pay back fast with regular updates.
  • Faster problem solving — a problem usually points to the last small change.
  • Faster customer feedback — features ship as soon as ready; feedback drives improvements.
  • A/B testing — deploy new version to some servers, divert traffic via load balancer, measure.

Infrastructure as Code (IaC)

Instead of manually configuring servers, write the infrastructure as a machine-processable model. CM tools like Puppet and Chef install software and services automatically.

Four characteristics

Visibility (a model anyone can read), Reproducibility (same sequence, same result), Reliability (no manual mistakes), Recovery (versioned — revert to a known-good config).

Containers

A container provides a stand-alone execution environment running on top of an OS like Linux. With Docker, a Dockerfile defines your software infrastructure as code; processing it builds an executable image.

  • Identical environments — same image for test and production.
  • Easy updates — rebuild the image with the new software, start it alongside the old, then divert traffic.

DevOps Measurement

Four types of software development measurement:

  • Process — development, testing, and deployment data.
  • Service — performance, reliability, customer acceptance.
  • Usage — how customers actually use the product.
  • Business success — how the product contributes to the business overall.
IBM's metrics scorecard (9 metrics)

Process: deployment frequency, change volume, lead time, percentage of failed deployments, mean time to recovery. Service: performance, availability, customer complaints, customer-number growth.

Chapter 4

Microservices Architecture

Software Services

A software service is a component accessed remotely over the Internet. Given an input, it produces an output — without side effects.

  • Accessed only through its published interface; implementation is hidden.
  • Stateless — state is either in a database or passed by the caller.
  • Because there's no local state, services can be reallocated or replicated freely.
Remember — modern web services

"Big" web services (SOAP, WSDL, XML) were too heavy. Modern services use lighter-weight protocols for lower overhead and faster execution.

Microservices

Small-scale, stateless services with a single responsibility, combined to create applications. They are fully independent — each with its own database and UI code.

Use case

For cloud-based products that need to be adaptable, scalable, and resilient, design around a microservices architecture.

Characteristics of Microservices

  • Self-contained — manage their own data and UI; no external dependencies.
  • Lightweight — low-overhead communication protocols.
  • Implementation-independent — any language, any database tech.
  • Independently deployable — each runs in its own process, deployed via automation.
  • Business-oriented — implement business capabilities, not just technical functions.

Cohesion, Coupling & Single Responsibility

A well-designed microservice has high cohesion and low coupling.

High Cohesion

  • All parts needed to deliver the service live inside the service
  • Internal pieces are closely related

Low Coupling

  • Few relationships with other components
  • Easy to change without breaking others
Remember

Each service should do one thing only — but "one thing" is judgement. It need not be a single function — it can be a single responsibility area.

Microservices Architecture (vs Monolithic)

An architectural style that solves two pains of monolithic systems:

  • The whole system must be rebuilt, retested, redeployed on every change — slow, and changes can affect unrelated components.
  • To handle more load, you must scale the entire system — even if only a few features are hot.
Microservices fix

Each service deploys in its own container, stopped/restarted in isolation. To handle more load, just scale out the hot service — cheaper than scaling up a monolith.

Example: Photo Printing System

A mobile app where users upload photos (from phone or Instagram), choose size and medium (mug, T-shirt, etc.), pay (Apple/Android Pay or saved card), and the print ships to their home.

  • Registration service
  • Authentication service
  • Upload service
  • Payment service
  • Printing service
  • Despatch service
API Gateway

The mobile app talks to a single API gateway, which routes requests to the right microservice — hiding internal layout from the client.

Decomposition Guidelines

  • Balance fine-grain vs performance — single-function services limit change scope but add inter-service comms (which slow things down).
  • Common closure principle — things that change together belong in the same service.
  • Align with business capabilities — each service supports a discrete area of business functionality.
  • Minimal data access — give each service only the data it needs; propagate shared data carefully.

Synchronous vs Asynchronous Communication

Synchronous (A waits for B)

  • A sends request, suspends
  • Waits for B's reply
  • Simple — but blocks

Asynchronous (A and B run concurrently)

  • A queues the request
  • A keeps working
  • B queues the result; A polls/receives later

Direct vs Indirect Communication

  • Direct — services know each other's address and call directly.
  • Indirect — services send to a message broker (a.k.a. message bus); the broker finds a service that can handle the request.
Remember

Indirect = looser coupling and easier load distribution; Direct = simpler but tighter coupling between services.

Data Design & Eventual Consistency

Isolate each service's data; if you must share, prefer read-only. Full ACID transactions across microservices are impractical.

  • Dependent inconsistency — one service's failure leaves another's data stale.
  • Replica inconsistency — concurrent replicas update their own DB copies independently.
Eventual consistency

Guarantee that databases will become consistent — implemented via a pending-updates log that replicas pull from and acknowledge.

Service Coordination: Orchestration vs Choreography

Most user sessions are workflows — a sequence of operations in a specific order (e.g. login with 3 attempts before failure).

Orchestration

  • A controller service drives the workflow
  • Calls login, then password, etc.
  • Central authority

Choreography

  • Services react to events
  • No central controller
  • Loose, event-driven coordination

Failure Management & Circuit Breakers

Three failure types in microservice systems:

  • Internal — service detects and reports (e.g. invalid URL).
  • External — outside cause makes the service unresponsive.
  • Performance — service degrades; needs external monitoring.
Circuit breaker pattern

Plain timeouts slow the whole system because every call to a dead service waits. A circuit breaker immediately denies access to a failed service — like an electrical breaker — avoiding the timeout delay entirely.

RESTful Services

REST = Representational State Transfer. The server transfers representations of digital resources to clients. Each resource has a unique URI.

  • HTTP verbs — GET, PUT, POST, DELETE map to operations.
  • Stateless services — no internal state retained between calls.
  • URI-addressable hierarchical resources (e.g. /incidents/A90/stonehaven/north/1).
  • JSON or XML for representations (JSON is most common).

RESTful CRUD Operations

Create / Read

  • POST → create new resource (error if it exists)
  • GET → read; never modifies state

Update / Delete

  • PUT → modify existing resource (not for creation)
  • DELETE → make inaccessible via URI
Remember

GET must be idempotent — repeating it (with no intervening PUT) always returns the same value.

Service Deployment & Continuous Pipeline

Each microservice team picks its own language, database, and libraries — so there's no single deployment config. Teams own deployment as well as development, using continuous deployment.

  • Commit triggers unit tests
  • Pass → build test system
  • Pass → integration tests
  • Pass → containerize, deploy, run acceptance tests
  • Pass → replace current service
Remember — versioned services

Keep older versions live behind the API gateway so you can roll back instantly if a new version misbehaves.
